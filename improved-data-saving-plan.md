# Document Overview
This is a 1-2 page document meant for human readability. It should provide a broad overview of the plan and its constraints and priorities. It is not an implementation plan. Once this plan is approved, it will be handed off to a new AI agent, which will use it to implement a change in MagScope.

# Plan Overview
The plan is to improve the tracking data saving in MagScope. This will affect both saving to disk and saving to RAM.

# Current Data Saving
Tracking data is currently generated by the video processing worker as each batch of images is processed in `videoprocessing.py`.

In RAM, the tracking results are written into a fixed-size shared-memory ring buffer (`TracksBuffer` in `datatypes.py`). The main purpose of storing the data in RAM is so it can be used for real-time plotting. It is also used for a few other real-time features, such as correcting stage or focus drift. It is not necessary to keep a complete session record in RAM, but the more detail retained, particularly for recent data, the better.

On disk, tracking data is currently saved as timestamp-named text files when acquisition saving is enabled. Each saved batch is written as a matrix with columns for time, X, Y, Z, bead ID, and ROI position. These files are fragmented.

A typical use case for the program would be 2-100 beads or ROIs per frame at 100 fps. A high-end use case would be 10,000 beads or ROIs and/or 10,000 fps.

# Why Improvements Are Needed
- The current system generates far too many files when saving to disk. This makes the data harder to post-process and harder to move around. It is also confusing for inexperienced users.
- The current system stores values as text files, which creates more data than is necessary.
- At high frame rates or with large numbers of beads, data saving can become a limiting factor for processing throughput.

# New Data Saving
For each saved frame, the tracking data should record the frame timestamp together with the per-bead tracking results. The frame timestamp should be stored once per frame as a `uint64`, representing nanoseconds since the Unix epoch.

For each bead detected in that frame, the saved data should include:
- the bead ID, stored as a `uint16`
- the bead position as three values (`x`, `y`, and `z`), stored as `float64`
- (optional) the ROI position as two pixel coordinates (`ROI-x` and `ROI-y`), each stored as a `uint16`

Saving the ROI-positions should become optional (default disabled). When enabled then each bead will also need two ROI-position values in pixels.

## Disk
Tracking data saving to disk should be stored in HDF5 extendable chunked datasets.

## RAM
The total RAM usage for this feature should be <1GB.
